# Life Signal Inbox 生活信号收件箱

`Life Signal Inbox`（生活信号收件箱）是一个计划整理项目，它把微信群通知、短信、课程公告、社区报名、截图文字这类碎片生活信息，整理成可执行的待办事项、日历事项和待确认项。

## 1. 项目痛点说明

生活里的关键信息经常藏在一大段通知里：截止时间、集合地点、需要携带的材料、联系人、费用、报名链接。用户看过不等于记住，也很难判断哪些信息还缺失、哪些时间会冲突。

这个项目不做普通聊天框，而是做一个生活消息处理台：输入通知后，系统先结构化，再验证，再进入待办清单。

## 2. 产品功能说明

- **全新 UI 架构 (Woven & Weft 质感)**：摒弃冷硬的 SaaS 风格，采用亚麻纹理、缝线边框和手写印染等“数字手账”材质，带来极具温度的沉浸式产品体验。
- **待办清单页**：左上角显示当前时间，进度条采用实体织线填色效果，下方展示提取后的“撕裂纸条”风格待办事项。
- **四象限规划页**：按重要/紧急程度划分为带有柔和浸染底色的十字缝线画布，支持像钉大头针一样拖动事项调整优先级。
- **信号分拣台 (原 AI 助手)**：支持文本口语交流和图片/截图上传。采用手账账本与便签贴纸的视觉隐喻。
- **多模态输入与双重引擎**：
  - 支持配置 **大语言模型 (Text)** 和 **视觉大模型 (Vision)**。
  - 支持上传图片后直接交由视觉大模型进行一步到位的多模态理解。
  - 当视觉模型严重翻车时，系统会自动降级（Fallback）调用本地 `PaddleOCR` 提取纯文本，再由文本大模型进行二次抢救。
- **强制确认的 Harness 验证**：自动检测缺失字段、低可信度字段和时间冲突。发现问题会在前端用红色缝线样式“挂起”，强制用户人工确认补齐后才能加入日历，彻底杜绝 AI 幻觉污染数据。
- **精准的跨期逻辑**：完美支持跨越多天的超长事件（如长假、连日展会），无论在“今天”还是“本周”的筛选标签下都能精准浮现。
- 支持保存到本地 SQLite 数据库、导出 `ICS` 日历文件、查看透明的 `JSON` 结构化数据。

## 3. Harness Engineering 工程化约束与反馈

本项目把 `Harness Engineering`（工程化约束与反馈）分成两层来说明：挑战要求更偏向“开发过程中如何用规则、工具和反馈循环约束 AI”，产品运行时也保留了同样的链路，方便演示。

### 3.1 开发过程中的 Harness

#### 上下文管理 / Agent Skill

开发时没有让 `AI`（人工智能）只根据一句话自由生成，而是先把规则写清楚：

- 提供 `AGENTS.md`（Agent 规则文档）指令，约定运行环境、沟通方式、敏感信息安全、提交格式和默认工作方式。
- 使用 `README.md`（项目说明文档）、`docs/harness-design.md`（Harness 设计说明）和 `docs/development-log.md`（开发过程记录）持续记录产品目标、边界和实现取舍。
- 把核心需求固定成明确流程：输入通知 -> 结构化提取 -> 自动验证 -> 待确认项编辑 -> 保存待办 -> 导出日历。
- 把后端拆成提取、验证、`OCR`（文字识别）、存储、`ICS`（日历文件格式）导出几个独立模块，减少 AI 修改代码时的上下文混乱。

#### 外部工具调用 / Tool 和 MCP

本次开发把这些 MCP 服务器作为 Agent 的外部工具层使用，同时配合本地库和 `CLI`（命令行工具）突破模型限制：

- `server-memory`（知识图谱记忆服务）：用于保存实体、关系和观察，帮助保留重要上下文。
- `server-sequential-thinking`（顺序思考服务）：用于把复杂问题拆成步骤，减少直接跳到实现的风险。
- `server-time`（时间服务）：用于确认当前时间、时区和日期计算。
- `context7-mcp`（文档检索服务）：用于查询库文档和代码示例。
- `mcp-server-fetch`（网页抓取服务）：用于请求网页或接口，获取外部数据。

- `PowerShell`（Windows 命令行）：启动服务、检查进程、执行脚本。
- `git`（版本控制工具）：分阶段提交功能改动，避免一次性混在一起。
- `pytest`（Python 测试框架）：验证后端提取、验证、存储、导出逻辑。
- `npm test`（前端测试命令）和 `npm run build`（前端构建命令）：验证前端逻辑和打包结果。
- `react-doctor`（React 检查工具）：检查前端结构、可维护性和潜在问题。
- `Playwright`（浏览器自动化工具）：验证网页真实交互，例如四象限拖拽和待确认项编辑。
- `PaddleOCR`（本地文字识别库）：把真实截图转换成文本。
- `SQLite`（轻量数据库）：保存历史事项，避免演示数据只存在前端临时状态。

也就是说，这里没有把大模型当成唯一能力来源，而是让它配合本地工具、测试框架和命令行反馈完成开发。

#### 验证与反馈循环

开发时的 `Feedback Loop`（反馈循环）是：

```text
明确规则和数据结构
-> 编写或补充测试
-> 实现功能
-> 运行后端测试、前端测试、构建和浏览器验证
-> 根据失败信息继续修正
```

本项目已保留开发记录和验证记录，详见：

```text
docs/development-log.md
docs/feedback-loop-log.md
```

### 3.2 产品运行时的 Harness

#### 上下文管理

后端会把每次输入整理成结构化上下文，而不是直接让模型返回一段总结。

上下文至少包含：

- `raw_text`（原始文本）
- `source_type`（来源类型）
- `current_date`（当前日期）
- `timezone`（用户时区）
- `recognized_items`（已识别事项）
- `historical_items`（历史事项）

当前版本的 `AI`（人工智能）提取模块已经独立成 Harness 管道：先构造结构化上下文，再要求模型输出 `JSON`（结构化数据格式），再进入自动验证。未配置模型时，系统会使用本地规则兜底。

#### 外部工具调用

- 图片上传双引擎：优先调用配置的**视觉大模型 (Vision Model)** 进行多模态理解；如果视觉模型未配置或提取严重失败，Harness 拦截器会触发 Fallback，调用本地 `PaddleOCR` 获取文本，再交由文本大模型处理。
- 日历导出：后端把历史待办导出为 `ICS`（日历文件格式）。
- 本地保存：历史事项保存在本地 `SQLite`（轻量数据库）。
- 调试展示：前端可以直接查看完整 `JSON`（结构化数据格式）结果，以及 Harness 的校验报告和错误拦截记录。

#### 验证与反馈循环

系统不会把 `AI`（人工智能）结果直接当最终答案使用，而是会自动检查：

- 每个事项是否有标题。
- 时间是否能解析成明确日期。
- 是否缺少地点、材料、联系人。
- 是否和历史事项存在时间冲突。
- 是否存在低可信度字段。

发现问题后，界面会显示风险提示和待确认项；用户确认后再加入待办清单。

如果模型输出不是合法 `JSON`（结构化数据格式），或者修正后仍不可用，系统会自动回退到本地规则提取，并在调试结果里记录兜底原因。

用户也可以在界面里补齐待确认字段，例如时间、地点、材料、联系人；补齐后调用后端统一验证接口重新检查。

## 4. 系统架构说明

项目采用前后端分离：

- 前端：`React`（前端框架） + `Vite`（前端构建工具），实现三页产品界面。
- 后端：`Python`（编程语言） + `FastAPI`（Python 后端框架），提供提取、验证、保存、导出接口。
- 数据：本地 `SQLite`（轻量数据库）保存历史事项。
- 可替换模块：提取模块、`OCR`（文字识别）模块、验证模块、日历导出模块都独立封装。

主要目录：

```text
backend/app/extractor.py      AI 提取适配层
backend/app/ai_extractor.py   AI Harness 提取管道
backend/app/validator.py      自动验证与反馈
backend/app/ocr.py            OCR 工具接口
backend/app/ics.py            ICS 日历导出
backend/app/storage.py        SQLite 本地历史保存
frontend/src/App.jsx          三页网页端界面
frontend/src/taskUtils.js     前端状态计算
```

运行后会生成本地数据库：

```text
backend/data/life_signal.db
```

这个文件只保存本机演示数据，不提交到仓库。

## 5. 如何运行项目

先启动后端：

```bash
cd backend
..\.venv\Scripts\python.exe -m pip install -r requirements.txt
..\.venv\Scripts\python.exe -m uvicorn app.main:app --reload --port 8000
```

如果已经激活 `.venv`（虚拟环境），也可以直接使用：

```bash
python -m pip install -r requirements.txt
python -m uvicorn app.main:app --reload --port 8000
```

启用真实 `AI`（人工智能）提取，需要设置环境变量：

```bash
set LIFE_SIGNAL_AI_PROVIDER=openai-compatible
set LIFE_SIGNAL_AI_API_KEY=你的模型服务密钥
set LIFE_SIGNAL_AI_MODEL=你的模型名称
set LIFE_SIGNAL_AI_BASE_URL=https://api.openai.com/v1
```

不设置这些变量时，系统仍然能运行，会自动使用本地规则兜底。

再启动前端：

```bash
cd frontend
npm install
npm run dev
```

打开：

```text
http://127.0.0.1:5173
```

运行测试：

```bash
cd backend
..\.venv\Scripts\python.exe -m pytest tests -q

cd frontend
npm test
npm run build
```

## 7. 完成范围说明

本版本优先保证核心演示闭环：

- 输入通知文本、上传图片、直接粘贴截图。
- `AI`（人工智能）或规则提取结构化事项；图片场景支持视觉模型优先、本地 `PaddleOCR`（本地文字识别库）兜底。
- 自动验证质量，并生成待确认项。
- 在信号分拣台逐项修改事件、时间、地点、备注，再单项加入待办清单。
- 支持手动添加待办、修改、删除、完成/取消完成。
- 支持重复规则、提醒、四象限分类、持续天数。
- 支持今日/逾期/本周待办看板和搜索筛选。
- 支持日历总览与四象限规划。
- 保存到本地 `SQLite`（轻量数据库）待办清单。
- 导出 `ICS`（日历文件格式），包含 `VALARM`（日历提醒）。

暂不包含真实账号体系、真实短信/微信接入、真实大模型服务和多用户同步。
