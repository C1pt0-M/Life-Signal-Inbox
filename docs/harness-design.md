# Harness Engineering 设计说明

这份文档用于说明本项目如何体现 `Harness Engineering`（工程化约束与反馈）。

这里的理解是：挑战要求主要看“开发过程中如何用工程约束、工具和反馈循环来驾驭 AI”，不是只要求产品界面里出现一个聊天机器人。为了让演示更直观，本项目同时把这些思想体现在产品运行链路里。

## 1. 开发过程中的 Harness

### 上下文管理

开发时没有让 `AI`（人工智能）自由发挥，而是先把任务拆成明确边界：

- Agent 规则：遵循用户提供的 `AGENTS.md`（Agent 规则文档）指令，约定 Windows11 运行环境、中文沟通方式、提交格式、安全要求和默认工作方式。
- 文档上下文：用 `README.md`（项目说明文档）、`docs/development-log.md`（开发过程记录）和本文件持续记录产品目标、功能范围和 Harness 设计。
- 产品目标：生活消息处理台，不做普通聊天框。
- 核心流程：输入通知、提取事项、自动验证、编辑确认、保存待办、导出日历。
- 数据结构：事项必须包含标题、时间、地点、材料、联系人、原文依据、可信度、四象限分类。
- 代码边界：提取、验证、`OCR`（文字识别）、存储、`ICS`（日历文件格式）导出分别封装。

这相当于先把 `AI`（人工智能）需要遵守的 `Agent Skill`（Agent 技能/规则）写清楚，让它围绕产品目标、结构化输出和安全边界工作，而不是只生成自然语言总结。

### 外部工具调用

开发过程中使用了多类工具来突破模型本身限制：

- `server-memory`（知识图谱记忆服务）：用于保存实体、关系和观察，帮助 Agent 保留关键上下文。
- `server-sequential-thinking`（顺序思考服务）：用于把复杂任务拆成步骤，先规划再修改。
- `server-time`（时间服务）：用于确认当前时间、时区和日期计算。
- `context7-mcp`（文档检索服务）：用于查询库文档和代码示例。
- `mcp-server-fetch`（网页抓取服务）：用于请求网页或接口，获取外部信息。
- `PowerShell`（Windows 命令行）：启动服务、检查端口和执行本地命令。
- `git`（版本控制工具）：按功能分类提交，保留清晰修改历史。
- `PaddleOCR`（本地文字识别库）：把截图转换成可处理文本。
- `SQLite`（轻量数据库）：保存历史事项，避免只停留在前端临时状态。
- `ICS`（日历文件格式）导出：把事项接到真实日历生态。
- `pytest`（Python 测试框架）：验证后端提取、验证、存储、导出。
- `node --test`（Node 测试工具）：验证前端状态计算和编辑逻辑。
- `react-doctor`（React 检查工具）：检查前端结构、可维护性和潜在问题。
- `Playwright`（浏览器自动化工具）：验证页面真实交互，例如四象限拖拽和待确认项编辑。

这些 `MCP`（模型上下文协议）服务器属于开发环境中的外部工具层；产品代码本身不依赖它们运行。开发过程主要依靠 MCP 工具、本地库、`CLI`（命令行工具）和自动化测试来获取真实反馈。这些工具让项目不是“模型说了算”，而是用真实工具检查结果。

### 验证与反馈循环

开发过程里的反馈循环主要有三层：

1. 写测试：先用测试描述期望行为。
2. 实现功能：让代码通过测试。
3. 运行验证：后端测试、前端测试、构建、`react-doctor`（React 检查工具）、浏览器自动化。

例如：

- `AI`（人工智能）提取模块先写测试，覆盖 `JSON`（结构化数据）解析、验证失败后修正、模型失败后规则兜底。
- 待确认项编辑先写前端工具函数测试，再接入页面。
- `SQLite`（轻量数据库）存储先写跨实例持久化测试，再替换原本的 `JSON`（结构化文件）存储。

这就是开发时的 `Feedback Loop`（反馈循环）：AI 生成或修改代码后，系统用测试和工具反馈问题，再继续修正。

## 2. 产品运行时的 Harness

产品本身也保留了同样的思想，方便演示。

### 上下文管理

用户输入不会直接进入一段聊天总结，而是会整理成结构化上下文：

```text
raw_text        原始文本
source_type     来源类型
current_date    当前日期
timezone        用户时区
recognized_items 已识别事项
historical_items 历史事项
```

`AI`（人工智能）提取时会收到这份上下文，并被要求输出结构化 `JSON`（结构化数据）。

### 外部工具调用

产品运行时使用的外部工具包括：

- 上传截图后调用 `PaddleOCR`（本地文字识别库），再把识别文本传入 `AI`（人工智能）结构化提取管道。
- 保存事项时写入 `SQLite`（轻量数据库）。
- 导出日历时生成 `ICS`（日历文件格式）。
- 查看调试信息时展示完整 `JSON`（结构化数据）。

这些工具让产品具备实际处理能力，而不是只依赖聊天回答。

### 验证与反馈循环

产品运行时的反馈循环是：

```text
输入通知
→ OCR 或文本输入
→ AI / 规则提取
→ 自动验证
→ 生成待确认项
→ 用户补齐字段
→ 重新验证
→ 保存待办 / 导出日历
```

验证规则包括：

- 是否有标题。
- 时间是否能解析成明确日期。
- 是否缺少地点、材料、联系人。
- 是否存在时间冲突。
- 是否存在低可信度字段。

如果模型输出不合法，系统会自动回退到本地规则提取，并在调试结果中记录原因。

截图上传已经接入完整链路：`PaddleOCR`（本地文字识别库）只负责把图片转成文字，后续仍然会进入 `AI`（人工智能）/规则提取、自动验证和待确认项编辑，而不是停留在纯文本识别。

## 4. Demo 中可以这样讲

可以用下面这段话解释：

```text
这个项目不是做一个聊天机器人，而是用 Harness Engineering 的方式约束 AI。
开发时，我先定义结构化上下文、输出格式、测试和验证规则，再让 AI 辅助实现。
产品运行时，通知会先变成结构化上下文，AI 只负责按 schema 提取，之后系统会自动验证。
如果缺时间、地点、材料或联系人，系统不会直接相信 AI，而是生成待确认项，让用户补齐后重新验证。
截图 OCR 后继续进入 AI 结构化提取，SQLite 保存和 ICS 导出则负责把结果保存和带到日历里，保证产品可以处理真实生活信息。
```
