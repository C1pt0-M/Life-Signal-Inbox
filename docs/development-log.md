# 开发过程记录

## 2026-04-19 OCR 接入

本次把截图文字识别从模拟结果改成了本地 `PaddleOCR`（本地文字识别库）。

### 主要改动

- `backend/app/ocr.py`（文字识别适配器）改为真实本地识别。
- 使用懒加载：只有用户上传图片时才初始化 `PaddleOCR`（本地文字识别库），避免后端启动变慢。
- 上传图片会先写入临时文件，识别完成后立即删除。
- 识别结果统一整理为：
  - `text`（识别出的完整文本）
  - `confidence`（平均可信度）
  - `blocks`（逐行文本块）
  - `provider`（识别来源）
- 保留失败兜底：库未安装或运行失败时，接口返回错误信息，不影响整个后端启动。

### 版本选择

实测 `paddleocr 3.4.1`（文字识别库）搭配 `paddlepaddle 3.3.1`（运行框架）在 Windows CPU 环境会触发：

```text
ConvertPirAttribute2RuntimeAttribute not support [pir::ArrayAttribute<pir::DoubleAttribute>]
```

改为 `paddlepaddle 3.2.0`（运行框架）后，本地截图 OCR 可以正常运行。

### 当前默认配置

默认使用轻量模型：

```text
PP-OCRv5_mobile_det
PP-OCRv5_mobile_rec
```

并关闭：

```text
doc orientation classify
doc unwarping
textline orientation
```

原因是本项目主要处理微信截图、课程通知截图、社区公告截图，不需要复杂文档矫正，优先保证演示稳定和响应速度。

### 已验证

- 单元测试覆盖结果解析、临时文件清理、库缺失兜底。
- 使用用户提供的任务截图实测，`PaddleOCR`（本地文字识别库）可以识别出中文正文，平均可信度约 `0.93`。

### 后续可优化

- 增加图片预处理，例如放大、灰度化、锐化。
- 对竖排文字、倾斜截图开启方向分类。
- 把模型路径固定到项目配置，避免不同机器重复下载。

## 2026-04-20 AI Harness 提取管道

本次把原来的规则模拟提取升级为可配置 `AI`（人工智能）结构化提取。

### 主要改动

- 新增 `backend/app/ai_extractor.py`（AI Harness 提取管道）。
- 模型输入不再是一段散文本，而是包含原文、来源、当前日期、时区、历史事项的结构化上下文。
- 模型输出被要求为 `JSON`（结构化数据格式），字段和现有待办数据结构一致。
- 模型结果会先进入自动验证；如果出现高风险问题，会带着验证结果再请求模型修正一次。
- 如果模型输出不是合法 `JSON`（结构化数据格式），系统会自动回退到本地规则提取，并记录兜底原因。
- 新增 `/api/config`（配置接口），前端会显示当前是 `AI Harness`（AI 工程化管道）还是规则兜底。

### Harness 设计对应关系

- 上下文管理：`build_context`（构造上下文）输出固定结构，再交给模型。
- 外部工具调用：支持兼容 `OpenAI`（模型服务）风格的接口，同时保留本地 `PaddleOCR`（文字识别库）。
- 验证与反馈循环：`validate_items`（验证事项）发现问题后，驱动模型修正；失败则规则兜底。

### 已验证

- 新增 AI 提取单元测试，覆盖结构化上下文、JSON 解析、验证修正、失败兜底。
- 后端全量测试通过。

## 2026-04-20 待确认项编辑

本次把验证结果从“只提示问题”升级为“可以直接修正问题”。

### 主要改动

- 新增 `/api/validate`（重新验证接口），前端编辑后复用后端统一验证规则。
- 待办清单页新增“待确认项编辑”面板。
- 支持编辑标题、时间、地点、材料、联系人。
- 用户手动编辑字段后，前端会把该事项可信度提升到可执行阈值，再交给后端重新验证。

### Harness 设计对应关系

- 验证与反馈循环：模型/规则提取后自动验证；验证失败后生成待确认项；用户补齐字段后再次验证。
- 上下文管理：编辑后的结构化事项仍保持同一数据结构，不回退成自然语言。
- 外部工具调用：重新验证通过后，事项仍可进入 `ICS`（日历文件格式）导出和本地保存流程。

### 已验证

- 后端新增接口测试。
- 前端新增编辑工具函数测试。
- 浏览器自动化实测：制造缺字段事项，补齐时间、地点、材料、联系人后，评分从低分变为 `100`。

## 2026-04-20 SQLite 本地存储

### 主要改动

- 新增 `SQLiteHistoryStore`（SQLite 历史事项存储）。

- 保存时写入可查询字段：标题、状态、保存时间、开始时间、结束时间、地点、来源、可信度、四象限。
- 同时保存完整事项 `JSON`（结构化数据），避免后续字段扩展时丢数据。
- `history`（历史接口）、`todos`（保存接口）、`export.ics`（日历导出接口）统一从 SQLite 读取。

### 已验证

- 存储层测试覆盖跨实例持久化。
- API 测试覆盖保存、读取历史、导出 `ICS`（日历文件格式）。

## 2026-04-20 截图 OCR 到 AI 提取闭环

本次把截图上传从“只识别文字”改成了“识别后自动理解并生成待办”。

### 主要改动

- 新增 `/api/ocr-extract`（截图识别并提取接口）。
- 后端流程改为：上传图片 -> `PaddleOCR`（本地文字识别库）识别文字 -> 构造结构化上下文 -> `AI`（人工智能）/规则提取 -> 自动验证。
- 前端“上传截图并提取”会直接展示待办事项、风险提示和待确认项，同时保留截图识别摘要，方便检查 OCR 原文。

### Harness 设计对应关系

- 上下文管理：截图文字不会停留在散文本，而是进入统一结构化上下文。
- 外部工具调用：`PaddleOCR`（本地文字识别库）负责突破模型不能直接读本地图片的问题。
- 验证与反馈循环：OCR 后的结果仍然走自动验证，缺时间、地点、材料、联系人时会生成待确认项。

## 2026-04-20 AI 模型 .env 配置

本次补充了后端 `AI`（人工智能）模型配置方式。

### 主要改动

- 新增 `backend/.env.example`（后端环境变量示例文件）。
- 新增 `.env`（环境变量文件）加载逻辑，后端启动时会读取项目根目录 `.env` 和 `backend/.env`。

- 不提交真实 `.env`（环境变量文件）。
- 不在前端保存 `API Key`（接口密钥）。
- 不硬编码模型密钥，优先使用环境变量。

## 2026-04-20 课表截图 OCR 后处理

本次针对课表截图补充了专用后处理，避免把低质量 OCR 文本硬拆成多个错误待办，并尽量按课程生成日程事项。

### 主要改动

- 新增课表截图识别逻辑：当 OCR 文本包含“我的课表”“第几周”“周一到周日”和多个课程名时，识别为课表类截图。
- 对课表类截图按课程名生成事项，例如“程序设计基础”“高等数学I”。
- 尝试从课表顶部的学年、学期、月份和日期行推导具体日期。
- 原始 OCR 文本继续作为 evidence（原文依据）保留。
- 对课程日程不再强制要求材料和联系人。
- AI 提示词新增约束：低质量 OCR 或表格网格文本不要强行拆成多个事项。

### 当前限制

课表截图属于网格信息，OCR 后可能丢失节次和行列关系。当前版本可以生成课程事项和日期，但具体节次仍建议人工核对原图。后续如果要精确生成每节课的开始时间，需要接入更适合表格/版面结构的 OCR 或图像理解模型。

## 2026-04-20 AI 助手固定格式输出

本次把 AI 助手的回复从“提取了几个事项”改成固定格式展示。

### 主要改动

- AI 助手回复统一显示：
  - 事件
  - 时间
  - 地点
  - 备注
- 联系人默认不在助手回复里展示，避免课表、课程通知这类场景出现不必要字段。
- 备注里优先展示课程/材料和可信度；没有材料时展示原文依据摘要。
- 前端使用固定模板渲染结构化结果，不依赖模型自由生成回复格式。

### Harness 设计对应关系

- 上下文管理：后端仍输出结构化 JSON。
- 验证与反馈循环：前端固定模板减少展示层不稳定。
- 外部工具调用：OCR/AI 提取结果进入同一展示格式。

## 2026-04-20 手动添加待办

本次把基础待办能力前置，用户不依赖 AI 也可以直接创建事项。

### 主要改动

- 待办清单页新增“手动添加待办”表单。
- 支持填写事件、日期、开始时间、结束时间、重复规则、地点、备注和四象限分类。
- 重复规则支持：不重复、每天这个时段、每个工作日、每个节假日。

- 手动添加的事项会直接保存到本地 `SQLite`（轻量数据库）待办清单。

### Harness 设计对应关系

- 上下文管理：手动事项也使用和 AI 提取事项一致的数据结构。
- 外部工具调用：保存仍走 `SQLite`（轻量数据库），导出仍走 `ICS`（日历文件格式）。
- 验证与反馈循环：前端先做基础必填检查，再由后端保存和导出。
