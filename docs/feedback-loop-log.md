# 反馈循环记录

这份文档记录项目开发中使用过的验证方式，方便说明 `Feedback Loop`（反馈循环）不是只写在概念里，而是实际参与了开发。

## 2026-04-20 当前验证方式

### 后端验证

```text
cd backend
..\.venv\Scripts\python.exe -m pytest tests -q
```

覆盖内容：

- 通知文本提取。
- 截图 OCR 后自动进入结构化提取。
- `AI`（人工智能）结构化输出解析。
- `.env`（环境变量文件）模型配置读取。
- 前端提交运行时模型配置，且接口不返回密钥。
- 模型输出不合法时的规则兜底。
- 待确认项重新验证。
- `SQLite`（轻量数据库）保存和读取。
- `ICS`（日历文件格式）导出。
- `OCR`（文字识别）结果解析和失败兜底。

### 前端验证

```text
cd frontend
npm test
npm run build
```

覆盖内容：

- 四象限分类计算。
- 用户拖拽后的象限覆盖逻辑。
- 待确认项编辑字段整理。
- 页面基础渲染，避免出现空白页。
- 前端生产构建是否可通过。

### 交互验证

开发过程中使用过 `Playwright`（浏览器自动化工具）检查真实页面交互：

- 四象限事项可以拖动到其他区域。
- 待确认项补齐后可以重新验证。
- 前端控制台没有关键报错。

### 质量检查

开发过程中使用过 `react-doctor`（React 检查工具）检查前端结构。当前主要剩余建议是页面状态较集中，属于后续可优化项，不影响 MVP 演示闭环。

### MCP 辅助

开发环境中配置了多个 `MCP`（模型上下文协议）服务器，可作为 Agent 的外部工具层：

- `server-memory`（知识图谱记忆服务）：保留关键上下文。
- `server-sequential-thinking`（顺序思考服务）：拆解复杂任务。
- `server-time`（时间服务）：确认时间和日期。
- `context7-mcp`（文档检索服务）：查询库文档。
- `mcp-server-fetch`（网页抓取服务）：获取网页或接口数据。

这些工具主要参与开发过程，不是产品运行所必需的依赖。

## 反馈循环如何工作

```text
AI 或人工修改代码
-> 运行测试/构建/浏览器验证
-> 根据失败信息定位问题
-> 修正代码或文档
-> 再次运行验证
```

这个流程对应挑战要求中的 `Feedback Loop`（反馈循环）：AI 不是直接给出最终结果，项目会用测试、构建和浏览器验证来发现问题，再继续修正。
